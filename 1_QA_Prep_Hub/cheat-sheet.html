🌱 Fundamentals

🧪

Software Testing

A process of evaluating a software application to detect defects and ensure it meets specified requirements.

Goal: Verify product behaves as expected & validate it solves the user's need.
🎯

Verification vs Validation

Verification: "Are we building the product right?" (reviews, walkthroughs)
Validation: "Are we building the right product?" (actual testing).

Example: Code review = Verification • UAT = Validation.
🛡️

QA vs QC

QA: Process-oriented, prevents defects.
QC: Product-oriented, identifies defects.

Tip: QA → "How we work". QC → "What we deliver".
🐞

Error / Defect / Bug / Failure

Error: Mistake by developer.
Defect: Deviation from requirement.
Bug: Informal name for defect.
Failure: System unable to perform required function.

Flow: Error → Defect → Bug → Failure.

🔄 SDLC & STLC

🏗️

SDLC Phases

Software Development Life Cycle — structured process for building software.

Phases: Requirement → Design → Development → Testing → Deployment → Maintenance.
🧭

STLC Phases

A sequence of activities performed during the testing process.

Phases: Requirement Analysis → Test Planning → Test Case Design → Env Setup → Test Execution → Test Closure.
📐

SDLC Models

Different approaches to organize the development & testing flow.

Common: Waterfall • V-Model • Iterative • Spiral • Agile (most used).
⚡

Agile & Scrum

Iterative methodology delivering working software in short cycles (sprints).

Ceremonies: Sprint Planning • Daily Standup • Sprint Review • Retrospective.

🧩 Types of Testing

✅

Functional Testing

Validates each function of the application works per requirement.

Example: Login with valid credentials → user lands on dashboard.
⚙️

Non-Functional Testing

Tests performance, usability, reliability — the "how" of the system.

Types: Performance • Load • Stress • Security • Usability • Compatibility.
🚬

Smoke vs Sanity

Smoke: Wide & shallow — build stability check.
Sanity: Narrow & deep — verify specific fixes work.

Trick: Smoke = "build accepted?" • Sanity = "fix verified?"
🔁

Regression Testing

Re-testing existing features after a change to ensure nothing is broken.

Example: After adding "Forgot Password", re-test full login flow.
♻️

Retesting

Executing the same test case again on a fixed defect to confirm resolution.

Diff: Retest = same case on fixed bug • Regression = related areas.
📦

Black / White / Grey Box

Black: No code knowledge.
White: Full code knowledge.
Grey: Partial code knowledge.

Who: Black → QA • White → Devs • Grey → Both.
🙋

UAT (Alpha & Beta)

Final testing by end-users before release.

Alpha: Internal users • Beta: Real customers.
🧭

Exploratory & Ad-hoc

Unscripted, intuition-driven testing to find unexpected defects.

Monkey testing: Random inputs, no rules — stress-test stability.

📑 Test Artifacts

📝

Test Case vs Scenario

Scenario: One-line "what to test".
Test Case: Detailed steps, data, expected & actual results.

Fields: ID, Title, Pre-conditions, Steps, Expected, Actual, Status.
📋

Test Plan

Document defining scope, approach, resources, schedule of testing.

Owner: Test Manager / Lead.
🔗

RTM

Requirement Traceability Matrix — maps requirements ↔ test cases to ensure full coverage.

Why: Detects missed requirements quickly.
🐛

Bug Life Cycle

New → Assigned → Open → Fixed → Retest → Verified → Closed (or Reopened).

Severity = impact • Priority = urgency.

🧠 Test Design Techniques

📏

Boundary Value Analysis

Test the edges of input ranges — most bugs hide at boundaries.

Range 1–100: Test 0, 1, 2, 99, 100, 101.
🧩

Equivalence Partitioning

Divide inputs into valid/invalid groups; test one value from each.

Age 18–60: Pick 25 (valid), 10 (invalid), 70 (invalid).
🗂️

Decision Table

Captures combinations of inputs & their expected outputs.

Use: Business rule-heavy modules like loan approval.
🔀

State Transition

Tests how the system moves between states based on events.

Example: ATM PIN attempts → 3 wrong = card blocked.

⚙️ Automation Basics

🤖

What is Automation Testing?

Using tools/scripts to execute pre-defined tests automatically and compare actual vs expected results.

Benefits: Speed • Reusability • Accuracy • CI/CD friendly.
⚖️

Manual vs Automation

Manual: Best for exploratory, UX, ad-hoc.
Automation: Best for regression, repetitive, performance.

Rule: Automate stable, repetitive flows.
🎯

When to Automate?

Stable features, large data sets, repetitive checks, cross-browser, smoke + regression suites.

Avoid: Frequently changing UI, one-off tests.
🏛️

Framework Types

Linear • Modular • Data-Driven • Keyword-Driven • Hybrid • BDD.

Most-used: Hybrid + BDD (Cucumber).

🌐 Selenium

🧪

Selenium WebDriver

Open-source tool to automate web browsers across languages (Java, Python, JS).

WebDriver driver = new ChromeDriver();
driver.get("https://qasync.app");
driver.findElement(By.id("login")).click();
📍

Locators

Strategies to find elements: id, name, className, tagName, linkText, partialLinkText, CSS, XPath.

Priority: id > name > CSS > XPath (XPath is slowest).
⏳

Waits

Implicit: Global wait for all elements.
Explicit: Wait for a specific condition.
Fluent: Explicit + polling interval + ignore exceptions.

new WebDriverWait(driver, Duration.ofSeconds(10))
  .until(ExpectedConditions.visibilityOfElementLocated(By.id("btn")));
🧱

Page Object Model (POM)

Design pattern — each page = a class with locators & methods. Improves reusability & maintenance.

Bonus: POM + PageFactory uses @FindBy for cleaner code.
🕸️

Selenium Grid

Run tests in parallel across multiple machines/browsers via Hub-Node architecture.

Use: Cross-browser regression at scale.

🧰 Test Frameworks & Tools

🧷

TestNG / JUnit

Java test runners with annotations, assertions, parallel execution & reporting.

@BeforeClass  @BeforeMethod  @Test(priority=1)
@AfterMethod  @AfterClass   @DataProvider
🥒

Cucumber (BDD)

Behavior Driven Development — write tests in plain English using Gherkin.

Feature: Login
  Scenario: Valid login
    Given user on login page
    When user enters valid credentials
    Then dashboard is displayed
📦

Maven

Build & dependency management tool for Java. Uses pom.xml.

Commands: mvn clean test, mvn install.
⚙️

Jenkins / CI-CD

Automate building, testing & deploying code on every commit.

Flow: Git push → Jenkins job → Run tests → Report → Deploy.
🌿

Git Essentials

Version control for tracking changes & collaborating.

git clone <repo>
git checkout -b feature/login
git add . && git commit -m "msg"
git push origin feature/login

🔌 API & Performance

🔗

API Testing

Verifying APIs at message layer — request, response, status, schema, performance.

Tools: Postman • RestAssured • Newman.
📡

HTTP Status Codes

2xx: Success • 3xx: Redirect • 4xx: Client error • 5xx: Server error.

Common: 200, 201, 301, 400, 401, 403, 404, 500, 503.
🧾

REST Methods

GET read • POST create • PUT update/replace • PATCH partial update • DELETE remove.

Idempotent: GET, PUT, DELETE.
🚀

Performance Testing

Load: Expected user volume.
Stress: Beyond limits.
Spike: Sudden load.
Soak: Long duration.

Tool: JMeter, k6, LoadRunner.

✨ Best Practices & Interview Tips

🌟

Automation Best Practices

  • Use POM + reusable utilities
  • Avoid hard-coded waits (use explicit waits)
  • Independent, atomic test cases
  • Externalize test data (JSON/Excel)
  • Integrate with CI for every commit
🩹

Handling Flaky Tests

Identify root cause: timing, env, data, locators. Add proper waits, isolate test data, retry strategically.

Don't: Blindly add Thread.sleep.
🎤

Interview Power Phrases

  • "I follow the STAR method to answer scenarios."
  • "I prioritize bugs by severity & business impact."
  • "I automate stable, high-ROI test cases first."
  • "Quality is everyone's responsibility — not just QA."
