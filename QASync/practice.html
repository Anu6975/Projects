Foundation

Build your QA fundamentals step by step

Software testing is the process of checking whether an application works as expected and meets user requirements without defects.
💡 Simple: Make sure your app behaves correctly before users touch it.

It helps identify defects early, ensures product quality, reduces future costs, and builds user trust.
💡 Simple: Fixing bugs early = less chaos later.

A test case is a set of steps, inputs, and expected results used to verify a specific feature of an application.
💡 Simple: A checklist to test one functionality.

A bug is an error in the software where the actual result differs from the expected result.
💡 Simple: When your app does something weird it shouldn’t.

Verification ensures the product is built correctly according to specifications, while validation ensures the right product is built based on user needs.
💡 Simple:

Verification = Are we building it right?
Validation = Are we building the right thing?

Manual testing involves executing test cases without using automation tools to identify defects.
💡 Simple: A human testing the app step by step.

Automation testing uses scripts and tools to execute test cases automatically, improving speed and accuracy.
💡 Simple: Letting tools do repetitive testing for you.

Black box testing focuses on testing the functionality of the application without knowing the internal code.
💡 Simple: You test like a user, not a developer.

White box testing involves testing internal code structure, logic, and paths of the application.
💡 Simple: You test the logic inside the code.

Regression testing ensures that new code changes do not negatively affect existing features.
💡 Simple: New changes didn’t break old stuff.

Smoke testing is a quick check to verify that the core functionalities of an application are working.
💡 Simple: “Does the app even start properly?”

Sanity testing is performed after minor changes to ensure that specific functionality works correctly.
💡 Simple: Quick check after a small fix.

A test plan is a document that defines the scope, strategy, resources, and schedule of testing activities.
💡 Simple: Blueprint of how testing will happen.

A test scenario is a high-level description of what needs to be tested from a user’s perspective.
💡 Simple: What user action you want to test.

A test scenario is broad, while a test case is a detailed step-by-step implementation of that scenario.
💡 Simple:
Scenario = Idea
Test Case = Detailed steps

STLC (Software Testing Life Cycle) is the sequence of phases in testing: requirement analysis, planning, design, execution, and closure. 💡 Simple: The full journey of testing.

SDLC (Software Development Life Cycle) is the complete process of software development from planning to deployment and maintenance. 💡 Simple: How software is built from start to finish.

Severity indicates how serious a defect is, while priority determines how urgently it should be fixed.
💡 Simple: Severity = Impact
Priority = Urgency

UAT is the final phase where end users validate whether the system meets business requirements.
💡 Simple: Client says “yes, this is what I wanted.”

A test environment is the setup (hardware, software, network) where testing is performed.
💡 Simple: The place where you test your app.
